## 2026-03-02 Redis 缓存优化

### 做了什么

给课程服务和订单服务做了 Redis 缓存防护，解决缓存穿透、击穿、雪崩三个问题，同时给购买课程接口加了分布式锁防止重复下单。

### 具体内容

**1. 布隆过滤器（防缓存穿透）**

- 新建 `CourseBloomFilterServiceImpl`，基于 Redis BitMap 实现布隆过滤器
- 服务启动时把数据库里所有课程 ID 预热进过滤器
- 在课程详情查询前加了判断，不存在的 ID 直接拒绝，不查缓存也不查数据库
- 新建课程时同步把新 ID 加入过滤器

**2. 分布式锁 + 双重检查（防缓存击穿）**
- 引入 Redisson，新建 `RedissonConfig`
- 课程详情缓存未命中时加 Redisson 分布式锁，只让一个请求回源数据库
- 锁内再检查一次缓存（双重检查），避免等锁期间已有其他线程填好缓存还重复查库
- Redisson 看门狗机制保证锁不会因业务执行慢而提前失效

**3. 随机 TTL + 定时预热（防缓存雪崩）**
- 所有缓存写入时 TTL 加随机偏移，避免大量 key 同时过期
- 新建 `HotCoursesCacheScheduler`，每 30 分钟主动刷新热门课程缓存
- 启动类加 `@EnableScheduling` 开启定时任务

**4. 防重复下单（order-service）**
- 购买课程接口加 Redisson 分布式锁，同一用户同时只能有一个下单请求执行
- 拿不到锁直接返回提示，不排队等待

### 改动的文件

- `course-service/pom.xml` — 加 Redisson 依赖
- `course-service/config/RedissonConfig.java` — 新建
- `course-service/service/impl/CourseBloomFilterServiceImpl.java` — 新建
- `course-service/service/impl/CourseServiceImpl.java` — 改造 getCourseDetail、getHotCourses
- `course-service/scheduler/HotCoursesCacheScheduler.java` — 新建
- `course-service/CourseServiceApplication.java` — 加 @EnableScheduling
- `order-service/pom.xml` — 加 Redisson 依赖
- `order-service/config/RedissonConfig.java` — 新建
- `order-service/service/impl/OrderServiceImpl.java` — createOrder 加分布式锁

## 2026-03-03 RocketMQ 消息队列接入

### 做了什么

引入 RocketMQ，将支付成功后「同步开通课程」改为「异步消息驱动」，解耦 order-service 和 learning-service。

### 具体内容

**1. 技术选型**
- 选用 RocketMQ（阿里巴巴出品），与项目已有的 Spring Cloud Alibaba 全家桶天然集成
- 使用 `rocketmq-spring-boot-starter 2.2.3`，兼容 RocketMQ Server 4.9.7

**2. 核心改造：支付成功开课链路解耦**

改造前（同步，强耦合）：
```
order-service 支付成功 → 直接调用本地 UserCourseService.activateUserCoursesFromOrder()
→ 写 user_course 表（同一事务，learning 挂了整个回调失败）
```

改造后（异步，解耦）：
```
order-service 支付成功 → 发 MQ 消息（Topic: ORDER_PAID）→ 立即返回
learning-service 监听 ORDER_PAID → 消费消息 → 写 user_course 表
```

**3. 消息设计**
- 消息体 `OrderPaidMessage`：包含 orderId、orderNo、userId、totalAmount、courseItems（含 courseId + purchasePrice）
- 放在 `common-service`，两个服务共用，不重复定义

**4. 幂等处理**
- learning-service 消费前先查 `user_course` 表是否已存在记录
- 已开通的跳过，防止消息重复投递导致重复开课

**5. 消息失败重试**
- 消费抛异常时 RocketMQ 自动重试，默认重试 16 次、间隔递增
- 超过重试上限后消息进入死信队列 `%DLQ%learning-consumer-group`，可人工处理

### 改动的文件

- `pom.xml`（父工程）— 加 `rocketmq-spring-boot-starter 2.2.3` 版本管理
- `common-service/mq/OrderPaidMessage.java` — **新建**，公共消息体
- `order-service/pom.xml` — 加 RocketMQ 依赖
- `order-service/application.yml` — 加 RocketMQ 配置（producer）
- `order-service/mq/OrderMessageProducer.java` — **新建**，消息生产者
- `order-service/service/impl/OrderServiceImpl.java` — `handlePaymentSuccess` 改为发 MQ
- `learning-service/pom.xml` — 加 RocketMQ 依赖
- `learning-service/application.yml` — 加 RocketMQ 配置（consumer）
- `learning-service/service/UserCourseService.java` — 加 `activateCoursesFromMessage` 接口
- `learning-service/service/impl/UserCourseServiceImpl.java` — 实现开课逻辑
- `learning-service/mq/OrderPaidMessageListener.java` — **新建**，消息消费者
- `deployment/docker-compose-local.yml` — **新建**，本地开发基础设施（Redis + RocketMQ + Nacos）
- `deployment/broker.conf` — **新建**，RocketMQ Broker 本地配置（inline 写入容器，避免 Windows Docker 挂载问题）

### 测试验证结果（2026-03-03）

```
18:24:28  order-service   创建订单 ORD202603031824280001（课程6，179元）✅
18:27:24  order-service   支付成功，MQ 消息发送至 Topic: ORDER_PAID ✅
18:27:44  learning-service 消费消息（线程: learning-consumer-group_1）✅
18:27:45  learning-service 幂等检查通过，INSERT user_course 写库成功 ✅
18:27:45  learning-service 事务提交，课程开通完成 ✅
```



引入 RocketMQ 实现订单异步解耦，解决支付链路强耦合问题。 原方案支付成功后同步调用 learning-service 开通课程，一旦下游服务超时或宕机整个支付回调即失败。改造后 order-service 支付成功仅需发送 MQ 消息即可返回，learning-service 通过 @RocketMQMessageListener 异步消费并开通课程，两个服务彻底解耦；消费前加幂等检查（查 user_course 表），防止消息重复投递导致重复开课；消费失败由 RocketMQ 自动重试（最多16次，间隔递增），超过重试上限进入死信队列（DLQ）做兜底，保障消息不丢失；同时该架构天然支持削峰，高并发下单时开课请求由 MQ 缓冲，下游按能力消费，避免流量尖峰击垮数据库。

**整条异步解耦链路验证通过。**

## 2026-03-04 数据库索引优化（核心表）

### 做了什么

针对核心业务表做索引补齐与联合索引优化，重点解决课程列表、订单列表、评论/笔记列表等高频接口在无索引情况下的全表扫描问题；使用 Navicat 执行 `EXPLAIN` 对比优化前后执行计划，确认从 `type=ALL`（全表扫描）切换为 `ref/range` 等索引访问方式。

### 具体内容

**1. 课程表（course）索引补齐**

- 课程列表常见筛选：`category_id + status` → 增加联合索引 `idx_category_status(category_id, status)`
- 讲师课程查询：`teacher_id` → 增加索引 `idx_teacher_id(teacher_id)`
- 热门课程：`status` 过滤 + `student_count` 排序 → 增加联合索引 `idx_status_student_count(status, student_count)`

**2. 订单表（order）用户维度查询优化**

- 用户订单列表：`user_id` → 增加索引 `idx_user_id(user_id)`
- 用户订单按状态过滤：`user_id + status` → 增加联合索引 `idx_user_status(user_id, status)`

**3. 评论/笔记列表索引补齐**

- 评论列表：`course_id + status` → 增加联合索引 `idx_course_status(course_id, status)`
- 笔记列表：`user_id + course_id` → 增加联合索引 `idx_user_course(user_id, course_id)`

**4. 学习记录与点赞表索引补齐**

- 学习记录按课程查询：`course_id` → 增加索引 `idx_course_id(course_id)`
- 评论点赞查询：`user_id + comment_id` → 增加联合索引 `idx_user_comment(user_id, comment_id)`

### EXPLAIN 验证要点

- 优化前：核心筛选 SQL 可能出现 `key = NULL` 且 `type = ALL`（全表扫描）。
- 优化后：`key` 命中新增索引，`type` 变为 `ref/range`，扫描行数（`rows`）显著下降。
- 例：`learning_record` 按 `course_id` 查询在命中 `idx_course_id` 后，`type=ref`，`ref=const`，预估扫描行数 `rows≈17`。
